  ---
  Conflict Resolution Strategy

  When a new fact's claim contradicts an existing fact in the same domain:

  1. Compare quality tiers: primary_url beats document beats secondary_url.
  2. Tie-break on recency: newer source wins within the same tier.
  3. Resolution: the loser gets superseded_by: new_fact_id, the winner gets
  supersedes: old_fact_id. Both stay in the file — nothing is deleted.
  4. Confidence adjustment: if sources disagree, cap confidence at 0.6 regardless of
  the winner's tier. Log it in open_questions.
  5. Changelog entry: write a fact_conflict_resolved entry with both versions, the
  resolution rule applied, and the source IDs.

  Example: URL says $49/mo (primary_url), PDF says $99/mo (document). Primary URL
  wins. PDF fact is marked superseded. Confidence drops to 0.6. Open question added:
  "Pricing conflict between website ($49) and internal doc ($99) — verify."

  This is explicit and auditable. Claude does not resolve conflicts via free-form
  reasoning — the rule is deterministic so it's testable and inspectable.

  Retrieval Strategy

  Step 1 — Should I retrieve at all?
  Check the last turn in session_history.json. If the question is clearly a follow-up
   to something already in context (e.g., "what about the enterprise plan?" after a
  pricing discussion), skip retrieval and answer from conversation context. The agent
   makes this call with a lightweight classification: a small Claude call with a
  [RETRIEVE | NO_RETRIEVE] decision prompt. Costs ~100 tokens but keeps the logic
  explicit.

  Step 2 — Which domains?
  Map the query to domains with a keyword → domain lookup table first (fast, free):
  "price", "cost", "plan", "tier", "billing"  →  pricing
  "feature", "product", "integration", "use case"  →  products
  "who", "founder", "CEO", "team", "employee"  →  team
  If no keyword hit, ask Claude to classify (small call). Return 1–3 domain names,
  not all 5.

  Step 3 — Token budget
  Estimate tokens per fact as len(claim) / 4 + overhead. Budget: 3,000 tokens for
  knowledge context. Sort facts by confidence × recency_weight, include top N that
  fit. Never dump all facts from a domain — only the relevant, high-confidence ones.

  Step 4 — Assemble
  Write selected facts to active_context.json. Include domain file name in the prompt
   so the agent can cite sources in its answer.

  Step 5 — Store new facts from conversation
  After every assistant turn, a lightweight pass checks if the user stated a new fact
   ("Actually, their enterprise plan is $500/mo"). If detected, Claude extracts the
  fact with domain + confidence, merges it into /knowledge, and logs it in the
  changelog. Source is recorded as "type": "conversation".

⏺ Good pressure-test. Here are the answers, in order.

  ---
  1. Deterministic vs Claude — Clear Separation

  Conflict resolution
  - Deterministic: Everything. Quality tier comparison (primary_url > document >
  secondary_url), recency tiebreak, confidence cap on conflict (0.6), marking the
  loser superseded=true. These are rules, not judgments.
  - Claude: Only contradiction detection — "do these two claims say different things
  about the same topic?" That's a semantic question a keyword check can't reliably
  answer. One small call, returns true/false + reason. Resolution itself never
  touches Claude.

  Retrieve vs no-retrieve
  - Deterministic: Handle with a simple heuristic chain (details in answer 4). No
  Claude call at all in the default path.
  - Claude: Not needed. Drop entirely.

  Domain selection
  - Deterministic: Keyword → domain lookup table. Fast, free, auditable.
  - Claude: Fallback only when zero keywords match. One call, returns a domain name
  from the fixed set. In practice this will rarely fire.

  Summary of Claude's actual responsibilities:

  ┌─────────────────────────────────────────┬────────┬───────────────┐
  │                  Task                   │ Claude │ Deterministic │
  ├─────────────────────────────────────────┼────────┼───────────────┤
  │ Extract facts from page/doc             │ ✓      │               │
  ├─────────────────────────────────────────┼────────┼───────────────┤
  │ Detect contradiction between two claims │ ✓      │               │
  ├─────────────────────────────────────────┼────────┼───────────────┤
  │ Classify domain (fallback only)         │ ✓      │               │
  ├─────────────────────────────────────────┼────────┼───────────────┤
  │ Detect new user-stated facts in chat    │ ✓      │               │
  ├─────────────────────────────────────────┼────────┼───────────────┤
  │ Answer questions                        │ ✓      │               │
  ├─────────────────────────────────────────┼────────┼───────────────┤
  │ Resolve conflict (who wins)             │        │ ✓             │
  ├─────────────────────────────────────────┼────────┼───────────────┤

      {
        "domain": "pricing",

        "losing_claim": "Starter plan is $39/month",

      }
    ]
  }

  Conflict history lives in the source file, not in every fact. Keeps facts lean and
  puts the conflict audit trail exactly where it belongs — in provenance.

  ---
  3. Facts That Don't Fit Predefined Domains

  Add a single misc.json catch-all domain. During extraction, if Claude can't map a
  fact to one of the five named domains, it assigns "domain": "misc".

  Retrieval handles it with one rule: always include misc.json as a secondary search
  target when the primary domain results are below a coverage threshold (e.g., fewer
  than 2 facts matched). No dynamic schema, no new domains, no schema changes
  mid-run.

  This means:
  - Extraction never fails silently — every fact lands somewhere.
  - Retrieval doesn't need special logic — misc.json is just another file.
  - open_questions.json remains separate from misc.json: open questions are things
  the system couldn't determine, misc facts are things it did determine but couldn't
  categorize.

  That's the entire fallback. No additional complexity.

  ---
  4. Simpler Retrieve vs No-Retrieve Heuristic

  Apply these checks in order. First match wins.

  NO_RETRIEVE if any of:
    - Turn is ≤ 4 words (greeting, acknowledgment: "thanks", "ok got it", "what?")
    - Turn contains: "you said", "earlier you", "just told me", "we discussed", "what
   did you say"
    - Turn starts with "actually" or "wait" (likely a correction, handle as
  store-new-fact path instead)
    - No word in the turn matches any domain keyword AND no word is in a
  stopword-excluded vocab

  RETRIEVE otherwise

  The fourth rule effectively means: if the question has no substantive content that
  maps to any domain, skip retrieval. The domain keyword table already exists for
  selection — reuse it for this check. If the question contains any keyword that maps
   to any domain, retrieve.

  No Claude call. No ambiguity. The only scenario where you'd want a smarter decision
   is a nuanced follow-up question, and in that case, retrieving unnecessarily costs
  you ~500 tokens in context — acceptable.

  ---
  5. Simplified Final Architecture

  What to drop entirely:
  - fact.id field
  - fact.supersedes field (keep only fact.superseded: bool)
  - fact.created_at (keep only updated_at)
  - Per-knowledge-file changelog array
  - Claude call for retrieve/no-retrieve
  - Claude call for domain selection (primary path; keyword table handles it)

  What stays:
  - Five named domains + misc.json + open_questions.json
  - Fact schema: claim, confidence, confidence_reason, source_id, updated_at,
  superseded
  - Source file: source_id, type, uri/content_ref, ingested_at, quality_tier,
  domains_updated, conflicts_resolved[]
  - active_context.json: rebuilt per turn, shows exactly what went into the prompt
  - session_history.json: hard window of last 8 turns, no summarization

  Simplified module responsibilities:

  knowledge.py     read/write facts, apply conflict resolution (pure deterministic)
  ingest.py        crawl/read → Claude extraction → call knowledge.py to merge
  retrieval.py     keyword domain selection, token budgeting, assemble active_context
  agent.py         heuristic retrieve decision → retrieval.py → Claude answer →
  detect+store new facts
  chat.py          CLI loop, session_history management

  The single clearest architectural line in the whole system:

  /working files are rebuilt every turn and can be safely deleted at any time — they
  are a window into the current session, not truth.
  /knowledge and /sources files are append-only with supersession — they are the
  truth. Nothing is ever deleted from them.

  That line, if held consistently in the code, directly satisfies the memory
  architecture criterion on its own.

  deterministic, never a Claude call.

  ---
  3. Decision Flows

  Ingestion

  INPUT: url or file/text
    │
    ├─ URL → crawl with requests+BS4 (deterministic)
    │         filter same-domain links → cap at 15 pages
    │         assign quality_tier: homepage = primary_url, subpages = secondary_url
    │
    └─ File/text → read raw text (deterministic)
                    assign quality_tier = document

  For each page/chunk of text:
    ├─ CLAUDE: extract_facts(text) → [{domain, claim, confidence, confidence_reason,
  open_questions}]
    │
    └─ For each extracted fact:
         ├─ Is domain in fixed set? (deterministic)
         │     NO → CLAUDE: classify_domain(claim) → domain name
         │           still no match → assign to "misc"
         │
         └─ Load existing domain file (deterministic)
              │
              └─ For each existing non-superseded fact with keyword overlap:
                   ├─ CLAUDE: detect_contradiction(new_claim, existing_claim) → bool
                   │
                   └─ If contradiction:
                        deterministic resolution:
                          compare quality_tier → winner keeps claim
                          loser: superseded=true, confidence_reason updated
                          both: confidence capped at 0.6
                          conflict logged to source provenance

  Append new fact to domain file (deterministic)
  Write source_NNN.json (deterministic)

  Key optimization: detect_contradiction is only called when two claims share at
  least one significant keyword (non-stopword). Avoids O(n²) Claude calls across
  unrelated facts.

  ---
  Retrieval and Answer Generation

  INPUT: user query
    │
    ├─ RETRIEVE DECISION (deterministic heuristic, no Claude):
    │    Skip retrieval if:
    │      - ≤ 4 words (greeting/ack)
    │      - contains: "you said" | "earlier" | "just told" | "we discussed"
    │      - no query word appears in domain keyword map AND query has no nouns
    │    Otherwise: retrieve
    │
    └─ If RETRIEVE:
         ├─ Keyword → domain mapping (deterministic lookup)
         │     No match → CLAUDE: classify_domain(query) → domain (rare fallback)
         │
         ├─ Load matching domain files + misc.json if primary < 2 facts
  (deterministic)
         │
         ├─ Filter: superseded=false only (deterministic)
         │
         ├─ Token budget: sort by confidence desc, accumulate until ~3000 tokens
  (deterministic)
         │
         └─ Write active_context.json (deterministic)

  Assemble prompt (deterministic):
    system_prompt + knowledge_context + last N session turns + user query

  CLAUDE: answer_question(assembled_prompt) → response citing domain file names

  Append turn to session_history.json, pop oldest if over window (deterministic)

  ---
  Store-During-Conversation

  After every user turn (runs in parallel with answer generation):
    │
    CLAUDE: detect_new_facts(user_turn) → [{claim, domain, confidence}] or []
    │
    └─ If facts detected:
         ├─ Same merge path as ingestion (deterministic)
         ├─ quality_tier = "conversation"  (lowest tier — any URL/doc will override)
         └─ source_id = "conversation_{session_id}_{turn_id}"
              written as a minimal source record to /sources

  This runs on the user's message only, not the assistant's response. If the user
  says "Actually their enterprise plan is $500/mo", Claude detects it, returns a
  structured fact, and knowledge.py merges it with quality_tier = "conversation". If
  a better source later contradicts it, the conversation fact loses
  deterministically.

  ---
  4. Implementation Order

  Ordered to get end-to-end working as fast as possible. Each step produces something
   runnable.

  Step 1 — config.py (~10 min)

  paths, window size (8). All constants in one place.

  Step 6 — agent.py (~15 min)

  1. Retrieve vs no-retrieve heuristic

  Sequential checks, first match wins — no Claude call:

  NO_RETRIEVE if:
    - query contains a conversational reference phrase:
      "you said", "you told", "earlier you", "just told me",
      "we discussed", "you mentioned", "as you said"
    - query is ≤ 4 words AND none of those words appear in KEYWORD_MAP
      (catches greetings/acks: "thanks", "ok got it", "what was that")

  RETRIEVE otherwise  ← default; missing context is worse than loading extra

  2. Domain routing

  1. Tokenize query to lowercase words + bigrams
  2. Look each token up in KEYWORD_MAP → collect matching domains
  3. If ≥ 1 match: use those domains
  4. If no match: single claude.classify_domain(query) call (rare fallback)
  5. After loading facts — if primary domains yield < 2 facts, append "misc" and
  reload

  3. Token budgeting

  Estimate each fact's size as len(json.dumps(fact)) // CHARS_PER_TOKEN. Walk the
  list (already sorted confidence-desc by load_facts_for_retrieval) and accumulate
  until the next fact would push past TOKEN_BUDGET. Set truncated: true if any facts
  were dropped.

  4. active_context.json shape

  {
    "assembled_at": "2026-04-18T12:10:00Z",
    "query": "What are the pricing tiers?",
    "retrieved_domains": ["pricing"],
    "skip_retrieval": false,
    "skip_reason": null,
    "facts_included": [
      {
        "domain": "pricing",
        "claim": "Starter plan is $49/month",
        "confidence": 0.85,
        "confidence_reason": "Explicitly on /pricing page",
        "source_id": "source_001",
